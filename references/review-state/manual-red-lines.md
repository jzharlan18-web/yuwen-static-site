# 人工审核红线（manual red lines）

本文件沉淀课件生成与发布过程中的硬约束。任何课件在 `current-review.md` 标记为「定稿完成」之前，都不得推送或声称已发布。

## 一、发布授权红线

- 唯一有效的发布指令是精确短语：**审核通过并发布**。
- 以下表达**不构成**发布授权：好的、可以、没问题、审核通过、OK、发吧、提交了没。
- 发布前必须：再读 `site-and-publishing.md` 与 `current-review.md`；确认远程地址为 `https://github.com/jzharlan18-web/yuwen-static-site.git`；确认无本任务清单以外的改动；只用明确路径暂存本课文件，禁止 `git add .` / `git add -A`；普通推送，禁止强制推送。

## 二、凭据红线

- 永远不索取、不回显、不写入、不过滤输出任何 GitHub Token、密码或密钥。
- 只使用操作系统钥匙串、GitHub CLI 已有安全登录或 WorkBuddy 已连接能力。
- 登录账号没有仓库写权限时停止，提示教师在本机完成安全授权。

## 三、内容保真红线

- 课文原句、题目、选项、原文证据不得杜撰或改写；逐字稿只给引文时，品读视图以所给引文为限。
- 不得虚构作者背景、课文原句或教学结论。
- 涉及真实学生答卷或成绩数据时，先匿名化，不展示可识别个人信息。

## 四、流程红线

- 生成后必须输出 `审核状态：待审核`，并说明「已核实 / 待补 / 内容检查 / 交付检查」。此时只能本地预览。
- 待补材料必须明确标注，不得自行补写。
- 不新增数据库、登录、服务器或框架，除非教师另行明确授权。

## 五、课堂白板接入红线（每次生成网页必验）

> 白板是全站共享工具（`tools/whiteboard/index.html`），经 `classroom-whiteboard.js` 弹窗打开。本课出现过的三个真实 bug（返回无效 / 更多·保存被挤出 / 自动保存刷屏）已修复，以下为防回归硬约束。

- **必须接入**：每课演示网页须加载站点 `assets/js/classroom-whiteboard.js` 并设 `data-lesson-title="<课文标题>"`；页面右下角须出现「白板」悬浮入口，点击能在新窗口打开白板并带回 `lessonTitle`/`lessonUrl`/`session` 参数。未接入视为不合格。
- **「返回」必须真正回到课件**：白板 `returnToLesson()` 在存在 `window.opener` 时，须 `window.opener.focus()` 后 `window.close()` 关闭白板弹窗；不得只 `focus()` 不关闭（否则白板仍盖在上层，看似“返回无效”）。验证方式：从本课「白板」按钮打开白板，点「返回《课文标题》」应关闭白板回到课件。
- **「更多 / 保存」必须常驻可见**：工具栏右侧「保存 / 全屏 / 隐藏栏 / 更多」须置于 `position:sticky;right:0` 的常驻组，窄屏横向滚动时不得被挤出视口（否则保存、导出等入口丢失）。修改白板工具栏后，须在 390px 手机宽与 1920px 一体机宽均确认「更多」按钮可见、可点。
- **自动保存必须静默**：自动保存只做防抖写入 localStorage，不得弹任何 toast；仅用户主动点「保存 / 本机保存」按钮时才弹“已保存到当前浏览器”确认。禁止每次操作弹「保存中…」「已保存」之类提示刷屏干扰上课。
- **共享文件改动须随课发布**：白板为全站共享工具，对其任何修改对所有课件生效；发布时若该文件有改动，须与本课文件一并按明确路径暂存提交，不得漏提、不得 `git add -A`。

## 六、投屏可读红线（字号与对比度）

> 演示网页定位为替代 PPT，用于 60 人教室一体机（1920×1080）投屏，最后排须可读。本红线防回归。

- **正文基准字号**：讲解/正文/卡片内容等主体文字，基准不得低于 `clamp(1.45rem, 3vw, 2rem)`（根字号 16px 时约 23–32px）；不得用 13–16px 小字做正文。
- **正文不小于栏目标题**：网页内各栏目标题（section h2，如「课题导入」「背景速览」等，渲染为 `clamp(1.45rem,3vw,2rem)`）即投影可读的下限参考；正文基准须 ≥ 该标题字号，不得更小。
- **关键内容更突出**：原文引用 `clamp(1.7rem,3.4vw,2.4rem)`（约 27–38px）；板书 `clamp(1.6rem,3.2vw,2.3rem)`（约 26–37px 衬线）；小标题 h3 `clamp(1.5rem,3vw,2rem)`；均须 ≥ 正文，形成层级。
- **层级不倒置**：栏目标题可更大，但正文不得为追求版面缩到手机阅读级小字；标签/导航/图注等次要文字可略小，但投影下仍须可辨（建议 ≥ 1rem）。
- **对比与遮挡**：正文与背景须保持足够对比度；不得用大面积动画、低对比或装饰遮字。
- **双宽目检**：新增/修改课件后，须分别在 1920×1080 一体机宽与 390px 手机宽目检文字清晰度；不清晰判不合格。

## 七、发布前账号校验红线（防“网页切了但实际没切”）

> 本次发布实战踩坑：教师在 GitHub **网页**切到 `jzharlan18-web`，但本机 `gh` 命令行的活跃账号仍是 `jerfasadinhu-ux`，导致 `git push` 被 403 拒绝。根因——**网页登录态（cookie）与 gh CLI 的 token 是两套独立认证，互不互通**。以下为防回归硬约束。

- **发布前必须 `gh auth status` 核实真实活跃账号**：收到「审核通过并发布」后、实际 `git push` 前，先运行 `gh auth status`，确认输出的 `Active account` 是具备本仓库写权限的账号（`jzharlan18-web`，token 含 `repo` scope 且为仓库 owner/协作者）。**不得仅凭教师口头“账号切好了”就推送**——须以命令输出为准。
- **账号切换只能走终端，网页切换无效**：若活跃账号不对或无写权限，须在本机终端执行 `gh auth login`（选 GitHub.com → HTTPS → Login with a web browser，浏览器授权时若恰是目标账号登录态则直接拿到其 token），登录后再 `gh auth switch --user <账号>` 切到目标账号；在 github.com 网页上切换登录**不会改变** gh CLI 的推送身份。
- **403 / Permission denied 立即停**：`git push` 返回 `remote: Permission to ... denied` 或 403 时，按凭据红线停止，不重试、不强制推送；本地提交保留，报告根因并给出重推指引（待教师用有权限账号授权后，普通 `git push origin main` 即可，无需重做课件）。
